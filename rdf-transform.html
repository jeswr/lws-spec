The JLWS core protocol [[JLWS-CORE]] is byte-native: a data resource is stored bytes plus a media type, and conformance never requires content parsing. This companion profile is the opt-in that makes a JLWS storage RDF-aware. A server declares, per source media type, which target media types it can derive through RDF-semantics-preserving transformation — advertised as ContentNegotiation capability entries in the storage description, pinned to this profile's semantics URI (https://w3id.org/jeswr/lws/transform/rdf-1). The profile fixes what the core leaves open: the round-trip contract (RDF 1.1 abstract-syntax equivalence, no inference), authoritative-bytes semantics with per-representation entity-tags and precondition rules that eliminate the lost-update ambiguity of existing Solid servers, a normalizes escape hatch for stores that cannot preserve bytes, the condition under which RDF PATCH formats may be advertised, and the hook by which indexes and query services may be enriched from parsed content. Finally, it defines how the Solid Protocol sits as a profile on top: "Solid-on-JLWS" is this profile plus a small mapping — the substrate never has to know RDF.

This document is a personal, experimental design draft, companion to [[JLWS-CORE]] and sharing its status: no standing, not a deliverable of any W3C group, unpublished, and awaiting review by its editor. Its namespace discipline is the core's (w3id.org/jeswr/lws; nothing minted under w3.org).

This document realises the "Personal Data Projection" requirement of [[LWS-UCS]] (requirement 33: "automatic transformation (projection) of personal data in formats consumable by non-LWS applications … without duplicating underlying resources") and extends the ContentNegotiation capability sketched in [[LWS10-CORE]] Discovery.

Introduction

Solid required every "RDF source" to be readable and writable as Turtle and JSON-LD and made RDF the storage's native model; LWS inverted this — resources are bytes, and the Working Draft merely notes that "Servers are free to support additional media types (e.g., text/turtle) through content negotiation" and sketches a ContentNegotiation capability ({source, target[]}) in its storage-description example. That sketch is the right layering but is not yet a contract: it does not say what a transformation preserves, which representation's bytes are authoritative, how entity-tags and preconditions behave across representations, or when RDF PATCH formats apply.

This profile supplies that contract. With it, an RDF-capable JLWS storage serves the same resource as Turtle to one client and JSON-LD to another, guarantees both are the same RDF graph, keeps writes safe under concurrency, and lets RDF-aware indexing and query be built on top — while a server that skips this profile remains fully conformant to the core, and a client can detect all of it from one capability entry.

This specification defines two conformance classes:

  • An RDF-transforming server — a JLWS server ([[JLWS-CORE]]) implementing this profile for at least one advertised source media type;
  • An RDF-aware client — a JLWS client that feature-detects this profile from the storage description before relying on any transformation.

Terminology

The terms storage description resource, "data resource", "capability", and "conformsTo" are used as defined by [[JLWS-CORE]]. "RDF graph", "RDF dataset", and graph isomorphism are defined by [[!RDF11-CONCEPTS]].

authoritative representation
The representation of a data resource in its stored media type: the byte stream most recently written (or its declared normalisation — ).
derived representation
A representation produced by the server from the authoritative representation through an advertised transformation.
RDF-readable resource
A data resource whose stored media type is an advertised transformation source (or target-reachable from one) under this profile.

Capability declaration

An RDF-transforming server MUST advertise, in the storage description's capability set, one ContentNegotiation entry per supported source media type: 

{
  "capability": [{
    "type": "ContentNegotiation",
    "source": "text/turtle",
    "target": ["application/ld+json", "application/n-triples"],
    "profile": "https://w3id.org/jeswr/lws/transform/rdf-1"
  }, {
    "type": "ContentNegotiation",
    "source": "application/ld+json",
    "target": ["text/turtle", "application/n-triples"],
    "profile": "https://w3id.org/jeswr/lws/transform/rdf-1",
    "normalizes": true
  }]
}
  • source — REQUIRED; the stored media type the entry applies to.
  • target — REQUIRED; the media types derivable from it. Each MUST be a concrete RDF serialisation of the same abstract syntax ().
  • profile — REQUIRED for entries claiming this profile; the value https://w3id.org/jeswr/lws/transform/rdf-1 pins the transformation semantics of . A ContentNegotiation entry without a profile (e.g. image/jpeg → image/png) makes no semantics-preservation claim and is outside this document.
  • normalizes — OPTIONAL, default false ().

The storage description's conformsTo array MUST include https://w3id.org/jeswr/lws/transform/rdf-1 when any entry claims this profile. Clients MUST NOT attempt transformation-dependent behaviour (RDF conneg, RDF PATCH, SPARQL query) against a storage that does not advertise it; servers MUST NOT advertise pairs they do not implement ([[JLWS-CORE]] conformance rule).

[[LWS10-CORE]]'s Discovery example writes this capability as {"type": "https://feature.example/ContentNegotiation", "source": …, "target": […]} — an unregistered example type with no semantics attached. This profile registers the term, keeps the source/target shape byte-compatible, and adds only the two members (profile, normalizes) that turn the sketch into a contract. (The design brief for this repository referred to the same mechanism as "ContentTransformation"; the Working Draft's term is retained for wire continuity — DECISIONS.md D18.)

Transformation semantics (the rdf-1 contract)

For every advertised source → target pair claiming this profile:

  • The derived representation MUST encode an RDF graph (or dataset, for dataset-capable pairs) isomorphic to that of the authoritative representation ([[!RDF11-CONCEPTS]] Section 3.6): same triples up to blank-node relabelling.
  • Transformation MUST NOT apply inference, schema expansion, or vocabulary rewriting: no triples added, none removed.
  • Relative IRIs MUST be resolved against the resource URI as base, identically in every representation.
  • For JSON-LD targets: when the authoritative representation is JSON-LD with a declared @context, servers MUST serialise derived JSON-LD using that context — subject to the no-dereference rule (): a remote context reference may be used only when it resolves from the server's allowlisted context cache. A compacted JSON-LD source declaring a remote context outside that allowlist cannot even be interpreted as an RDF graph without resolving that context, so the server MUST decline the transformation for that resource (406 + problem details, as for an unparseable source) — it MUST NOT resolve the context with a network fetch. Only when the source is context-independent — expanded form, or all of its context inline — may the server transform it without a cached context. When the source has no context (e.g. Turtle), the server MUST use a documented, stable context or expanded form. In every case clients MUST NOT depend on any particular compaction beyond the graph contract.
  • Blank-node identifiers are NOT stable across representations or requests; clients requiring stable references use IRIs.
  • If the authoritative representation fails to parse under its stored media type, the server MUST answer requests for derived representations with 406 Not Acceptable carrying a problem details body of type https://w3id.org/jeswr/lws/problems/unparseable-source — the stored bytes remain readable in the stored media type. (A byte store never breaks because a transform does.)

Authoritative bytes, entity-tags, and preconditions

This section removes the lost-update ambiguity that RDF-serving Solid servers exhibit when the same resource is read and written through different serialisations.

  • Stored bytes are authoritative. The byte stream accepted at write time is the resource's authoritative representation. A read in the stored media type MUST return those bytes exactly, unless normalizes is declared for that source type ().
  • Derived representations are RFC 9110 representations of the same resource — negotiated via Accept; responses vary by representation, so responses to requests subject to transformation MUST carry Vary: Accept.
  • Entity-tags are per-representation ([[!RFC9110]] Section 8.8.3: an entity-tag validates a representation). The ETag returned with a Turtle response and the ETag returned with a JSON-LD response of the same resource state MUST differ, and each MUST change whenever the resource state changes.
  • Write preconditions evaluate against the representation the ETag names. A server MUST accept an If-Match carrying either the authoritative representation's current ETag or the current ETag of the derived representation the client last read; both name the same resource state. Stale values of either MUST yield 412. The core's 428 rule applies unchanged: no unconditional overwrite of an existing RDF-readable resource.
  • Writes in a derived media type: a server MAY accept a PUT whose Content-Type is an advertised type reachable from the stored type, but ONLY when the written type is itself advertised as a source in a ContentNegotiation capability (bidirectional coverage). This condition is load-bearing: because on acceptance the written bytes become the new authoritative representation and the stored media type becomes the written Content-Type, accepting a write in a type with no source entry would strand the resource in a stored type the server cannot transform — silently breaking the negotiation targets it advertises. Servers MUST NOT silently transcode a write into a different stored type unless normalizes is declared. A server that does not accept writes in a given type (including the not-a-source case above) answers 415 with problem details naming the accepted types.

The normalizes flag

Some stores cannot preserve written bytes for a media type — e.g. a quad-store-backed server that parses on ingest and reserialises on read. Such a server MUST declare "normalizes": true on the affected source entry. Semantics:

  • Read-back in the stored media type is graph-isomorphic to what was written () but NOT byte-identical; clients MUST NOT rely on formatting, comment preservation, prefix labels, or blank-node identifiers.
  • The normalisation MUST be deterministic for a given resource state (two reads without an intervening write return identical bytes), so ETags remain coherent.
  • Without the flag (the default), byte-exact read-back is a MUST — the core's byte-native promise. Servers SHOULD prefer byte-preserving storage; normalizes is an honest escape hatch, not the recommended mode.

RDF PATCH formats (only when on)

RDF-level patching is meaningless on a byte substrate and is therefore scoped entirely to this profile:

  • A server MUST NOT advertise an RDF patch format (application/sparql-update, text/n3 for N3 Patch [[N3-PATCH]]) in Accept-Patch or in a PatchSupport capability entry for a media type unless this profile is on for that media type — mirroring the Working Draft's Discovery example, which pairs text/turtle with application/sparql-update in PatchSupport.
  • An RDF patch applies to the resource's RDF graph; the resulting graph is reserialised into the authoritative representation. Applying an RDF patch is a state change like any other: all representation ETags rotate, and the precondition discipline of applies to the PATCH request itself.
  • After an RDF patch, byte-exact preservation of the previous serialisation is by definition impossible for the changed portions; a server MAY preserve unaffected formatting but clients MUST NOT rely on it. (A server for which this matters SHOULD declare normalizes for patched types.)
  • Patch evaluation MUST be atomic and MUST enforce the same authorization as a content write; conditional patch semantics (e.g. an N3 Patch ?conditions that fails to match) yield 409 Conflict with problem details.

RDF-aware indexing and query (enrichment hook)

The core's search services derive types from Link rel="type" headers at write time, never parsing bodies ([[LWS-SEARCHINDEX-ED]]). When this profile is on, the server MAY additionally derive index entries from the parsed RDF content of RDF-readable resources (rdf:type statements and descriptive properties), ranked equally with header-derived entries — the searchindex draft's "MAY derive from content" hook, made well-defined by this profile's parse contract. The SparqlQueryService ([[JLWS-CORE]] Query services) MUST only be advertised when this profile is on, and its access-control semantics (per-solution authorization filtering) follow [[AC-SPARQL]]. All such derived views remain bound by the core's oracle-freedom invariants: live authorization filtering, caller-visible counts, immediate revocation effect.

Solid as a profile on top ("Solid-on-JLWS")

This section is an informative mapping sketch; a full Solid-on-JLWS profile would be its own companion document.

The layering thesis: JLWS is the substrate; Solid is a profile. A Solid-on-JLWS storage is a JLWS storage that:

  1. Turns this profile on for text/turtle and application/ld+json (both directions) — giving Solid's mandatory RDF-source conneg pair [[SOLID-PROTOCOL]];
  2. Serves container listings as it already does — the core's JSON-LD container representation is an RDF graph; with this profile on, a Turtle rendering of the listing gives LDP-accustomed clients a familiar containment graph without the server implementing LDP;
  3. Maps Solid-OIDC to the OIDC authentication suite — the WebID is the CID subject (a WebID document content-negotiates to a CID; solid:oidcIssuer becomes the jlws:OpenIdProvider service — the migration path in the editor's lws-keycloak notes), with the DPoP presentation profile ([[JLWS-CORE]] Proof-of-possession profiles) satisfying Solid-OIDC's DPoP expectation [[SOLID-OIDC]] for clients that need it;
  4. Maps WAC/ACP documents to ODRL grants — acl:Read → odrl:read; acl:Write → odrl:modify + jlws:create + odrl:delete; acl:Append → jlws:append; acl:Control → grant-management rights over the access services; acl:accessTo → a DataResource/Container target; acl:default → a recursive Container target; acl:agentClass foaf:Agent → the public assignee ([[WAC]]). The WAC↔ACP conversion precedent (the @solid/object wacToAcp/acpToWac accessors) shows such mappings are mechanically checkable;
  5. Advertises N3 Patch and/or SPARQL Update via PatchSupport () for Solid clients that write via PATCH.

Under this mapping, existing RDF client stacks consume a JLWS storage unchanged — content negotiation and parsing, typed reads over the graph, standard serialisation on write — with the capability entry telling the client, before its first request, whether RDF conneg will work. The substrate never learns RDF; everything RDF-shaped lives behind the one advertised profile. What Solid-on-JLWS deliberately does not restore: RDF-mandatory resources, slash-semantics containment (the core's containment is metadata with guaranteed path alignment — strictly more predictable), or client-managed containment triples.

Security and privacy considerations

  • Parser surface: transformation makes the server parse untrusted bytes. Parsers MUST be resource-bounded (document size, triple count, nesting depth, parse time); parse failures are per-resource errors (), never server faults.
  • No dereference during transformation: serialisation and parsing MUST NOT fetch remote documents — no JSON-LD remote @context resolution beyond a server-configured allowlist of cached contexts, no owl:imports, no follow-your-nose. A remote-context fetch performed on attacker-supplied content is an SSRF primitive; the core's SSRF policy ([[JLWS-CORE]] Server-side request forgery) applies to any allowlisted fetch.
  • Oracle discipline: content-derived index entries expose content through a side channel; the enrichment hook () is bound by the core's oracle-freedom rules, and index visibility MUST match resource read visibility per-agent.
  • Normalisation information loss: normalizes discards comments and formatting, which may themselves carry meaning for some workflows; the flag makes the loss visible in advance rather than discovered on read-back.
  • ETag correlation: per-representation ETags MUST NOT embed content digests that let an agent with access to one representation infer bytes of another it cannot read (relevant only under representation-scoped authorization extensions; with resource-scoped authorization, all representations share visibility).

Test-vector families

This profile's vectors ship in the shared suite ([[JLWS-CORE]] Appendix, Test-vector index; repository test-vectors/vectors/rdf-transform/, 17 cases): capability/conformsTo consistency; graph round-trips (Turtle ↔ JSON-LD, blank-node isomorphism, no-inference, resource-URI base resolution); the ETag rotation matrix across representations on write; If-Match acceptance with the derived representation's ETag; normalizes determinism; unparseable-source degradation (stored bytes still readable); RDF-PATCH-off rejection (415, no Accept-Patch advertisement without the profile); and the byte-native behaviour with the opt-in off. Still deferred (recorded in the suite's GAPS.md with reasons): bounded-parser rejection of depth/size bombs, declared-context reuse in JSON-LD compaction, derived-type writes, the RDF PATCH application tranche, and the WAC→ODRL mapping table as executable assertions (a Solid-on-JLWS companion deliverable).
