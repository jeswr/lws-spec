This document specifies a clean-slate core protocol for Linked Web Storage: a byte-native, self-describing, OAuth-2-authorised storage substrate for the Web, in which data storage, entity authentication, access control, and application provider are loosely coupled. It is designed as a wire-compatible superset of the W3C Linked Web Storage Working Group's Working Draft [[LWS10-CORE]]: it keeps that draft's sound mechanisms — opaque byte-bearing resources, containment expressed as metadata, JSON-LD container representations, RFC 9264 linkset metadata with strict conditional-request discipline, and an RFC 8693 token-exchange authorization chain terminating in short-lived audience-restricted Bearer tokens — and diverges only where the draft has identified gaps: RFC 9728 protected resource metadata replaces bespoke discovery; idempotent create is defined; containers and data resources are strictly separated; URI-path alignment is guaranteed; server-to-client notification bindings (SSE and WebSocket) join the signed-webhook binding; a WebAuthn authentication suite joins the OIDC and self-signed suites; proof-of-possession token presentation is available as an optional, negotiated profile; access requests and grants use a strict ODRL profile; and a security model covering SSRF and oracle-freedom is normative. RDF content transformation, query, and versioning are companion specifications advertised through the storage description; the core never requires content parsing.

This document is a personal, experimental design draft. It has no standing whatsoever: it is not a deliverable, work item, or position of the W3C Linked Web Storage Working Group or of any other group, and is not endorsed by W3C or by anyone else. It exists to explore, in complete form, a design that the Working Group's own drafts cover partially — and to be offered as input to that group only if and when its editor chooses to do so.

This draft is currently unpublished and under review by its editor. It mints no identifiers under w3.org; all vocabulary lives under the personal namespace https://w3id.org/jeswr/lws# (), and the working name JLWS is an explicit placeholder pending the editor's decision (see docs/OPEN-QUESTIONS.md).

Drafting disclosure: this draft was prepared with AI assistance (Claude Fable 5, Anthropic), working from the Working Group's published drafts and repository, the editor's own prior prototype prose (jeswr/lws-keycloak spec/authn.md and spec/authz.md), and the primary sources cited throughout, and is awaiting review by its human editor. Introduction

The W3C Linked Web Storage Working Group is re-standardising the Solid storage layer on a byte-native, JSON-LD-described, OAuth-2-authorised basis, dropping the RDF-mandatory / LDP heritage of the Solid Protocol [[SOLID-PROTOCOL]]. Its core protocol [[LWS10-CORE]] reached First Public Working Draft on 31 March 2026. That draft is directionally sound, but — by the Working Group's own resolutions and issue tracker — it defers or leaves open a set of load-bearing questions: there is no client-facing notification channel (server-to-server webhooks were resolved to come first); authorization-server discovery uses a bespoke as_uri parameter and /.well-known/lws-configuration path where RFC 9728 and RFC 8414 now provide registered equivalents; there is no idempotent create; the container-as-data-resource question is unresolved; there is no delegation, group, query, or versioning story against the Use Cases document's requirements [[LWS-UCS]]; and the security and privacy sections are largely unwritten.

This document is one editor's complete answer. It is deliberately a wire-compatible superset of [[LWS10-CORE]]: a client written against the Working Draft's container representations, linkset metadata, operations, authentication suites, and token-exchange authorization chain interoperates with a server conforming to this document, because those mechanisms are adopted as-is (with their requirements cited). The divergences are confined to the gaps listed above, each marked in place with a rationale and collected in .

Design principles

  1. Byte-native substrate. A resource is an opaque representation — bytes plus a media type. Conformance to this core never requires parsing resource content. All protocol-internal structure (container listings, the storage description, subscriptions, access documents) is JSON-LD-profiled JSON: plain-JSON clients parse it; RDF clients get triples for free.
  2. No bespoke duplicates of shipped IETF/W3C standards. Where a registered mechanism exists — RFC 9110/9111 semantics and caching, RFC 8288 links, RFC 9264 linksets, RFC 7386 merge patch, RFC 9457 problem details, RFC 9728 protected resource metadata, RFC 8414 AS metadata, RFC 8693 token exchange, RFC 9068 JWT access tokens, RFC 9396 rich authorization requests, RFC 9421/9530 message signatures and digests, [[CID-1.0]] identifier documents, Activity Streams 2.0 [[AS2-VOCAB]], ODRL 2.2 [[ODRL-MODEL]] — this specification uses it rather than minting a lookalike.
  3. Bearer-baseline authentication, proof-of-possession as a negotiated profile. The default token presentation is the deliberately simple chain the editor prototyped and carried into [[LWS10-CORE]]: token exchange producing an audience-restricted at+jwt with a recommended lifetime of 300 seconds or less, presented as a Bearer token. Sender-constrained presentation (DPoP [[RFC9449]], DPoP-SK [[DPOP-SK]]) is an optional profile negotiated through RFC 9728 resource metadata — never a reversal of the baseline.
  4. Capability-advertised everything. Content transformation, PATCH formats, resumable uploads, query tiers, notification channel bindings, versioning, and recursive delete are feature-detected from one signed, CID-shaped storage description whose conformsTo member carries protocol version URIs.
  5. Enforcement is internal; ODRL is the interchange. The server's policy engine is out of scope; access requests, grants, and consent receipts are exchanged as a strict ODRL profile.
  6. Fail closed, oracle-free. Existence is hidden from agents with no access; every derived view (listings, indexes, counts) is filtered by live authorization.
  7. Strict tree with guaranteed URI-path alignment. Containment remains expressed in metadata, but resource URIs are guaranteed to be path-aligned with their containment position, restoring relative-URI and object-store affordances.

Document family

This core document specifies the substrate only: resources and containers, metadata, operations, authentication, authorization, access requests and grants, discovery, and notifications. The following are companion specifications, each optional and advertised through the storage description ():

  • JLWS RDF Content Transformation Profile — the opt-in that makes a byte-native storage RDF-aware (drafted alongside this document);
  • JLWS Query Services (type index/search and access-controlled SPARQL) — planned;
  • JLWS Versioning — planned.

This specification defines the following conformance classes:

  • A JLWS server — an HTTP server conforming to all MUST-level requirements of sections through that are not scoped to an optional capability or profile.
  • A JLWS client — an HTTP client conforming to the client-scoped MUST-level requirements of the same sections.
  • A JLWS authorization server — an OAuth 2.0 authorization server conforming to .
  • A notification server — a JLWS server additionally conforming to for at least one subscription type it advertises.
  • A PoP-capable server — a JLWS server additionally conforming to for at least one advertised proof-of-possession presentation profile.

Optional features are conformant only when advertised in the storage description (); a server MUST NOT advertise a capability, service, or profile it does not implement.

This specification defines the following terms:

storage

storage root
The top-level container of a storage. It has no parent container.
resource
An HTTP resource within a storage; either a container or a data resource, never both ().
container
A resource whose role is to aggregate member resources. Its representation is the server-managed listing defined in .
data resource
A byte-bearing resource of any media type. The server stores and returns its representation without interpreting it (except under an advertised transformation capability).

authentication credential
A tamper-evident security token asserting claims about an agent, validated under an authentication suite (). Equivalent to the "end-user credential" of [[LWS10-CORE]].
authentication suite
A defined serialization and validation mechanism for a concrete class of authentication credential, identified by an OAuth token-type URI.
agent
The subject of an authentication credential: a person, organisation, or software actor identified by an absolute URI.
storage controller
An agent authorised to manage access to a storage (grant, deny, revoke).
access request
An ODRL document by which an agent asks a storage controller for access ().
access grant
An ODRL document by which a storage controller records granted access. A grant is a record, not a token ().
subscription
A registration expressing interest in notifications about resource changes ().
notification
A message describing an event that occurred on a resource.

Namespace and context

The JLWS vocabulary namespace is https://w3id.org/jeswr/lws#, abbreviated jlws:. The normative JSON-LD context URI is https://w3id.org/jeswr/lws/v1 (). Protocol conformance URIs are minted under https://w3id.org/jeswr/lws/protocol/.

[[LWS10-CORE]] uses https://www.w3.org/ns/lws# with context https://www.w3.org/ns/lws/v1 — which remains unpublished (w3c/lws-protocol#60). This personal draft MUST NOT mint or squat identifiers in the w3.org namespace, so it uses w3id.org/jeswr/lws with term-identical mappings: the term names (Container, DataResource, items, totalItems, …) and their JSON shapes match the Working Draft exactly, so a future one-line context substitution restores full wire identity. Additionally, wherever this specification requires a https://w3id.org/jeswr/lws# URI in a link relation or type position, implementations MUST also accept the corresponding https://www.w3.org/ns/lws# URI as an equivalent alias, preserving interoperability with Working-Draft implementations.

Resource model

Resource classes

Resources in a storage are classified as either a container or a data resource ([[LWS10-CORE]], Container Model). A data resource is byte-bearing and format-unconstrained: per [[!RFC9110]], this specification does not limit the nature of a resource, and a server MUST NOT require any particular content type or structure of data-resource content. A server MAY enforce media-type or size policy per storage ([[LWS-UCS]] requirement 1), and MUST advertise any such restriction as problem details on rejection.

Containers are not data resources

A resource MUST be exactly one of container or data resource. A server MUST NOT serve client-authored content as the representation of a container: the only representation of a container is the server-managed listing of . Descriptive content about a container is attached through its linkset resource (e.g. a describedby link to a separate data resource), keeping the container's ETag and content negotiation unambiguous.

Whether a container may simultaneously be a data resource is the Working Draft's most persistent unresolved contention (trait/mixin vs strict separation; ETag and content-negotiation ambiguity — w3c/lws-protocol#126, London F2F Day 1 minutes). This specification resolves it by fiat in favour of strict separation: one resource, one role, one unambiguous ETag. The describedby linkset pattern provides the trait model's expressiveness without its representation ambiguity. Recorded as DECISIONS.md D3; flagged for the editor's steer in OPEN-QUESTIONS.md.

Auxiliary resources are roles

Auxiliary resources (a resource's linkset, description, access documents) are not a distinct resource class: they are ordinary resources standing in a role toward another resource, discoverable through typed links (rel="linkset", rel="describedby", …). This adopts the model the Working Group resolved by vote 8–0–1 on 22 June 2026 (w3c/lws-protocol#167; minutes).

Containment

Containment is expressed in metadata, following [[LWS10-CORE]] Containment:

  • Servers MUST include a Link header with rel="up" pointing to the parent container in responses to GET and HEAD requests on any non-root resource.
  • A container's members are listed in the items property of its representation. The items list is server-managed: clients MUST NOT be able to modify it directly; membership changes occur only as a side effect of resource creation, deletion, and (where supported) move.
The Working Draft's Logical Resource Organization section states that "The server manages this list; clients cannot modify it directly", while its Metadata section lists up and items as client-manageable "Core Metadata". This specification resolves the drift in favour of server management: up and items are system-managed, read-only metadata (see ). Reparenting, where offered, is the explicit move operation, not a metadata edit. Recorded as DECISIONS.md D13.

The server MUST maintain containment integrity at all times ([[LWS10-CORE]], Containment Integrity):

  • Creation: when a resource is created in a container, the server MUST atomically add it to that container's items list.
  • Deletion: when a resource is deleted, the server MUST atomically remove it from its parent container's items list.
  • No orphans: every non-root resource MUST be reachable from the storage root through the containment hierarchy.
  • No cycles: a container MUST NOT directly or indirectly contain itself.
  • Single containment: every non-root resource has exactly one parent container. (Multiple containment is out of scope, matching the Working Group's resolution — London F2F Day 2, w3c/lws-protocol#135. If ever introduced, it would be an advertised capability, not a core change.)

URI-path alignment

Resource URIs MUST be path-aligned with containment:

  • The URI of every resource MUST be the URI of its parent container extended by exactly one non-empty path segment (for containers, followed by a terminating /).
  • Container URIs MUST end in /; data-resource URIs MUST NOT.
  • The storage root's URI is the prefix of every resource URI in the storage.

Containment semantics remain carried by metadata (rel="up", items): clients discover structure through metadata, and servers still assign final URIs at creation time. Path alignment guarantees that relative-URI resolution, object-store key mapping, and audience/realm containment checks () behave predictably.

The Working Draft states "clients SHOULD NOT assume that URI structure reflects containment" ([[LWS10-CORE]], Resource Identification). With multiple containment resolved out of scope for 1.0, the residual value of URI-independence is small, and its costs are real: loss of relative-URI affordances, ambiguous realm containment, and no object-store mapping (argued in w3c/lws-protocol#135). The editor's own prototype already assumes alignment: "For LWS implementations where lws:containment conforms with URI containment the realm will typically be an lws:StorageRoot" (jeswr/lws-keycloak spec/authz.md §4.1). A path-aligned server is wire-compatible with Working-Draft clients (they simply observe metadata that happens to match paths); the guarantee only constrains servers. Recorded as DECISIONS.md D4; steer question 3 in OPEN-QUESTIONS.md.

Container membership and authorization

If an agent has read access to a container, the complete container listing — the union of the items arrays across all pages, when the representation is paginated () — MUST include the identifiers of all member resources to which that agent has access; each individual paginated response carries only the current page's members. The representation MUST NOT include members the agent has no access to (see the oracle-freedom rules, ), and totalItems MUST count only the members visible to the requesting agent (across the whole collection, not the current page).

The Working Draft permits listings to include identifiers of resources the client cannot access ("It MAY also contain the identifiers for resources contained in that container to which the client does not have access" — [[LWS10-CORE]], Container Membership and Authorization). This specification forbids that: the searchindex Editor's Draft [[LWS-SEARCHINDEX-ED]] already adopts caller-visible-only counts for its derived views, and there is no reason the primary listing should be a weaker oracle than the index built over it. Recorded as DECISIONS.md D12.

Read access to a container listing does not imply access to the listed resources, and vice versa.

Container representation and pagination

This section adopts the Working Draft's container representation ([[LWS10-CORE]], Container Representation) unchanged in shape.

Properties

A container representation MUST include:

  • id — the container's URI;
  • type — the value "Container";
  • totalItems — an integer count of members disclosable to the requesting agent (); SHOULD be exact, and MAY be approximate only when the approximation is computed exclusively from the caller-visible view — never derived from an unfiltered membership total, which would leak the existence of hidden members ();
  • items — a flat array of member descriptions (empty array when empty; the current page only when paginated).

Each member description MUST include id and type ("DataResource" or "Container", or an array containing at least one of the two; additional user-defined types as URIs MAY be included). It SHOULD include size (bytes) and modified (ISO 8601), and MUST include mediaType for data resources.

The Working Draft's published shape is the flat items array; the repository's F2F artifact container.json and the searchindex Editor's Draft use an embedded ContainerPage/contains shape. This specification standardises the published flat shape and expresses pagination purely in Link headers. Recorded as DECISIONS.md D13.

Media type and content negotiation

Container representations (and the storage description) MUST be available as application/ld+json; profile="https://w3id.org/jeswr/lws/v1". Servers MUST also answer requests for application/lws+json, application/ld+json, and application/json with the identical payload bytes — only the Content-Type header varies ([[LWS10-CORE]], LWS Media Type, Content Negotiation). Servers MAY offer additional serialisations (e.g. text/turtle) via content negotiation; the RDF Content Transformation companion profile makes that support advertisable and precise.

Pagination

Servers SHOULD paginate container listings whose membership exceeds a server-determined threshold. Pagination is link-based ([[!RFC8288]]): paginated responses MUST carry rel="first", MUST carry rel="next" on all but the last page, MUST omit rel="next" on the last page, MAY carry rel="prev" and rel="last", and page URIs are opaque — clients SHOULD NOT construct or modify them. When paginated, items contains the current page while id, type, and totalItems reflect the whole (visible) membership. Paginated responses use status 200.

Metadata

This section adopts the Working Draft's linkset metadata model ([[LWS10-CORE]], Metadata) with the categories drift fixed.

  • All resource metadata is a set of typed links ([[!RFC8288]]) anchored at the resource.
  • For each resource, the server MUST make its metadata available as a standalone linkset resource per [[!RFC9264]], available as application/linkset+json, discoverable via a Link header with rel="linkset" on GET/HEAD responses.
  • Linkset updates MUST be atomic with the associated resource operations.
  • Deleting a resource MUST delete its linkset resource.

The Working Draft's metadata table places up/items in a client-manageable "Core Metadata" category while its resource-organization text forbids client modification of items. This specification keeps them system-managed (see ); the two-category model above replaces the three-category one. Recorded as DECISIONS.md D13.

Updating metadata

  • Servers MUST support PATCH on the linkset resource with application/merge-patch+json ([[!RFC7386]]) and MUST advertise it via Accept-Patch; PUT and additional patch formats are OPTIONAL and MUST be advertised via Allow / Accept-Patch when supported.
  • Linkset GET/HEAD responses MUST carry an ETag; successful updates MUST rotate it.
  • A PUT or PATCH on a linkset resource without If-Match MUST be rejected with 428 Precondition Required ([[!RFC6585]]); a stale If-Match MUST be rejected with 412 Precondition Failed.
  • Servers MUST reject attempts to modify system-managed links (e.g. with a problem details response of type https://w3id.org/jeswr/lws/problems/system-managed-metadata).

Operations (abstract)

JLWS defines five abstract operations, deliberately independent of any single transport binding (rationale: w3c/lws-protocol#24; loose coupling per [[LWS-UCS]] requirement 28). The HTTP binding in is the only binding defined here and is REQUIRED for conformance; other bindings may be defined by companion documents.

create resource
Adds a new resource (container or data resource) to an existing container. Inputs: the target container, an optional identity hint, content and type (for data resources), or an explicit target identifier (idempotent form). The server determines or validates the final identifier, atomically updates membership, and initialises system metadata.
read resource
Retrieves a data resource's representation or a container's listing (filtered by authorization), with metadata links.
update resource
Replaces or partially modifies a data resource's content (or a linkset's links), atomically, under optimistic concurrency control.
delete resource
Removes a resource and its metadata, updating parent membership atomically. Containers must be empty unless recursion is explicitly requested and supported.
move resource (OPTIONAL)
Reparents a resource into a different container. Under path alignment this assigns a new URI; the operation returns it. Advertised via the MoveResource capability. (An optional move was resolved into the Working Draft's direction at the London F2F — Day 2 minutes, resolving w3c/lws-protocol#83.)

HTTP binding

General requirements

  • Servers MUST implement HTTP semantics per [[!RFC9110]] and SHOULD implement caching per [[!RFC9111]].
  • Every GET/HEAD response for a storage resource MUST carry an ETag, the rel="linkset", rel="up" (non-root), and rel="type" links, and the storage-description link (). These entity-tags MUST be strong ([[!RFC9110]] Section 8.8.1): If-Match uses strong comparison ([[!RFC9110]] Section 13.1.1), so a weak validator can never satisfy the write-precondition discipline this specification mandates — a weak ETag would make the required conditional updates impossible for a conforming client.
  • All 4xx and 5xx responses MUST carry an [[!RFC9457]] problem details body (application/problem+json), except where HTTP semantics do not permit response content — a 4xx/5xx response to HEAD ([[!RFC9110]] Section 9.3.2) carries the same header fields (including Content-Type: application/problem+json) that the corresponding GET error response would, without the body; 304 is not an error and carries none.
    The Working Draft SHOULDs problem details. Uniform machine-readable errors are cheap for servers and load-bearing for autonomous clients; this specification requires them. A registry of JLWS problem types lives at https://w3id.org/jeswr/lws/problems/. Recorded as DECISIONS.md D17.
  • Storage-quota exhaustion MUST be reported as 507 Insufficient Storage.
  • Servers MUST reject request targets containing path-traversal sequences (.., %2e%2e, and equivalents) before any authorization matching (jeswr/lws-keycloak spec/authz.md §7.5).

Create

POST to a container (server-assigned name)

Per [[LWS10-CORE]] Create Resource:

  • Clients create a resource by POST to an existing container URI. The server assigns the final URI, optionally incorporating a Slug header hint; the server MAY decline the hint on conflict or naming policy.
  • To create a container, the client MUST include Link: <https://w3id.org/jeswr/lws#Container>; rel="type" (the w3.org/ns/lws#Container alias MUST be accepted — ). To create a data resource, the client includes body content with a Content-Type.
  • Clients MAY seed user-managed metadata via Link headers; server-managed metadata MUST be generated by the server and MUST NOT be overridable by client links.
  • On success the server MUST return 201 Created with the new URI in Location and the rel="up", rel="linkset", and rel="type" links. Resource, metadata, and membership creation MUST be atomic.
  • If the target container does not exist (or must be hidden), the server responds 404; on a naming conflict the server assigns a different URI or responds 409.

PUT with If-None-Match: * (idempotent create)

Servers MUST support creation by PUT to the desired resource URI carrying If-None-Match: * ([[!RFC9110]] Section 13.1.2):

  • The target URI MUST be path-aligned under an existing parent container (); the server MUST NOT auto-create intermediate containers, and MUST respond 409 Conflict (problem type …/problems/missing-parent) when the parent does not exist.
  • If the target already exists, the server MUST respond 412 Precondition Failed and MUST NOT modify the resource — making retries safe.
  • Creation via PUT MUST carry If-None-Match: *. An unconditional PUT (one carrying neither If-None-Match: * nor If-Match) is never a create: when its target URI does not identify an existing resource, the server MUST respond 428 Precondition Required ([[!RFC6585]]) rather than create the resource. Together with the 428 rule for unconditional PUT to an existing resource (), every PUT in JLWS is explicitly conditional.
  • A URI ending in / creates a container (the request body MUST be empty); otherwise a data resource is created from the body and Content-Type.
  • On success: 201 Created, with the same links and atomicity requirements as POST-create.
The Working Draft's only create is non-idempotent POST, and it acknowledges the duplicate-on-retry hazard in-spec ("POST is not idempotent. Repeating it may create duplicates"); idempotent create is an open Working Group issue (w3c/lws-protocol#173). PUT + If-None-Match: * is plain RFC 9110 — no new mechanism — and becomes well-defined here because path alignment makes client-chosen URIs meaningful. Recorded as DECISIONS.md D2.

Read

  • GET on a data resource returns its stored representation with the stored media type, unless the server advertises a ContentNegotiation capability for the stored media type () and the request's Accept header selects an advertised derived target — in which case the server returns the negotiated derived representation with its correct Content-Type and Vary: Accept, per the transformation's declared semantics (for RDF, the RDF Content Transformation companion). Absent that capability, or when Accept selects the stored media type, the stored bytes are returned. Servers MUST support range requests ([[!RFC9110]] Section 14) for data resources.
  • GET on a container returns the container representation ().
  • Servers MUST support HEAD, returning GET's headers without a body.
  • Servers MUST support conditional GET (If-None-Match; If-Modified-Since) answering 304 Not Modified when applicable.

Update

  • PUT replaces a data resource's content. If the server supports ETags for the resource (it MUST — ), it MUST reject an unconditional PUT to an existing resource (one carrying neither If-Match nor If-None-Match: *) with 428 Precondition Required, and a stale If-Match with 412 Precondition Failed. On success: 200 or 204 with the new ETag.
  • PATCH ([[!RFC5789]]) applies a partial modification. Servers MUST support application/merge-patch+json ([[!RFC7386]]) for JSON-family data resources and for linkset resources; support for further formats (per media type) MUST be advertised via Accept-Patch and the PatchSupport capability (). Content PATCH for non-JSON media types is undefined by this core; RDF patch formats are defined by the RDF Content Transformation companion and MUST NOT be advertised without it.
  • PUT/PATCH on a resource URI modifies content only, never the linkset, except under the explicit Prefer: set-linkset mechanism of [[LWS10-CORE]] Update Resource, which servers MAY implement.

Delete

  • DELETE removes a resource, its linkset, and its membership entry atomically; on success 204 No Content. The parent container's ETag MUST change.
  • DELETE on a non-empty container without recursion MUST be rejected with 409 Conflict. Servers MAY support recursive deletion; clients request it with Depth: infinity ([[!RFC4918]]), and servers advertising the RecursiveDelete capability MUST honour it, deleting the subtree atomically.
  • Clients MAY (and SHOULD, for guarded deletes) send If-Match; servers MUST honour it.

Move (OPTIONAL)

Servers advertising the MoveResource capability MUST implement move as follows: the client PATCHes the resource's linkset changing the up link target to the destination container (the one system-managed link writable through this operation, and only under this capability). On success the server MUST: (a) assign the resource a new URI path-aligned under the destination container, preserving the final path segment unless it conflicts; (b) atomically update both containers' membership; (c) respond 200 with the new URI in Location; and (d) SHOULD answer subsequent requests for the old URI with 308 Permanent Redirect for an implementation-defined period. Notifications () for a move are emitted as Delete(old)+Create(new) unless a Move activity is negotiated.

Authentication

This section adopts the Working Draft's suite architecture ([[LWS10-CORE]] Authentication; jeswr/lws-keycloak spec/authn.md) unchanged, ships three of its four suites, and adds a WebAuthn suite.

Authentication credential data model

An authentication credential MUST include tamper-evident claims:

  • subject (REQUIRED) — an absolute URI identifying the agent;
  • issuer (REQUIRED) — an absolute URI identifying the issuing entity;
  • client (REQUIRED) — an identifier for the client application; SHOULD be an absolute URI;
  • audience restriction (RECOMMENDED) — a list of values that SHOULD include an authorization server identifier.

A credential MUST be signed; asymmetric signatures are RECOMMENDED, and JWT-serialised credentials MUST NOT use the none algorithm. Each authentication suite MUST be associated with a token-type URI, preferably one registered in the IANA OAuth URI registry; authorization servers SHOULD advertise supported suites via subject_token_types_supported in their RFC 8414 metadata (SHOULD rather than MUST for the privacy reason recorded in ).

Identity documents (CID)

Agent identifiers dereference to controlled identifier documents [[!CID-1.0]]. In the absence of a pre-existing trust relationship, a verifier MUST dereference the credential's subject URI and obtain a valid CID whose id equals the subject identifier; suite-specific validation then binds the credential to keys (verificationMethod) or issuers (service) published in that document. The OIDC issuer binding is a CID service entry of type jlws:OpenIdProvider (accepting the lws#OpenIdProvider alias) whose serviceEndpoint equals the credential's issuer — the analogue of Solid's solid:oidcIssuer ([[LWS10-AUTHN-OPENID]], Validation).

All server-side dereferencing performed during validation is subject to the SSRF requirements of . Verifiers SHOULD cache CID-derived keys respecting HTTP caching metadata; issuers SHOULD serve CIDs with caching headers that make such caching effective (jeswr/lws-keycloak spec/authn.md, SSI-CID §7).

Authentication suites

Suites are optional, pluggable profiles; a conforming JLWS authorization server MUST implement at least one. The suite definitions below are normative summaries; the OIDC, self-signed-CID, and did:key suites match the corresponding Working Group suite drafts and those documents' fuller text applies.

OpenID Connect suite

Per [[LWS10-AUTHN-OPENID]]: the credential is a signed OIDC ID Token; sub/iss/azp carry subject/issuer/client; aud SHOULD include the client and any target authorization server. Validation: dereference sub to a CID; locate the jlws:OpenIdProvider service matching iss; perform OpenID Connect Discovery; validate the JWT per OpenID Connect Core Section 3.1.3.7. Token type: urn:ietf:params:oauth:token-type:id_token.

Self-signed CID suite

Per [[LWS10-AUTHN-SSI-CID]]: a self-issued signed JWT with sub == iss == client_id (an absolute URI), aud including the target authorization server, and exp/iat. Validation: dereference sub to a CID; select the verification method by the JWT kid per [[!CID-1.0]] Section 3.3; verify per [[!RFC7515]] Section 5.2. Token type: urn:ietf:params:oauth:token-type:jwt. This suite gives autonomous agents and services first-class identity without an IdP.

Self-signed did:key suite

Per [[LWS10-AUTHN-SSI-DID-KEY]]: as the CID suite, but sub is a did:key URI and the verification key is extracted from the identifier itself (did:key Method Section 3.1.3) — no dereference. Token type: urn:ietf:params:oauth:token-type:jwt.

WebAuthn suite

The credential is a WebAuthn authentication assertion over a server-issued challenge, packaged for presentation as an RFC 8693 subject_token exactly as specified by the wire contract of [[WEBAUTHN-REAUTH]] (its ./protocol module): the assertion binds the agent's registered passkey to the exchange request; the authorization server validates the assertion against the credential public key registered for the subject and MUST enforce challenge freshness, origin/RP-ID binding, and signature-counter or device-attestation policy per WebAuthn Level 3. Subject/issuer/client claims are carried in the token-exchange envelope defined there. Token type: the URI minted by that contract.

The Working Group ships a SAML 2.0 suite and no WebAuthn suite; its own drafting notes already prefer direct WebAuthn ("This is the preferred use of WebAuthn with Linked Web Storage — … it removes the need to trust an additional IDP server", jeswr/lws-keycloak spec/authn.md, WebAuthn note), and OpenID-suite guidance recommends providers support WebAuthn. This specification ships the passwordless suite users actually deploy and leaves SAML to be defined as an external suite by whoever needs it — the suite architecture makes that a profile, not a fork. Recorded as DECISIONS.md D7.

Client identity

Client identifiers SHOULD be absolute URIs dereferencing to an OAuth Client ID Metadata Document [[CLIENT-ID-METADATA]]: plain JSON, no JSON-LD context required. Issuers of authentication credentials are responsible for validating client identifiers and MAY use this mechanism, client ID prefixes, or OpenID Federation.

This replaces Solid-OIDC's JSON-LD Client Identifier Documents, following the editor's migration notes ("the client identifier document will respond with the content type application/json and without the need for JSON-LD support") and his upstream issue w3c/lws-protocol#38. Recorded as DECISIONS.md D8.

Authorization

The authorization chain is the editor's prototyped model (jeswr/lws-keycloak spec/authz.md), as carried into [[LWS10-CORE]] Authorization: authentication credential → RFC 8693 token exchange at a trusted authorization server → audience-restricted, short-lived RFC 9068 access token → presentation to the storage server. The single structural divergence is discovery: RFC 9728 protected resource metadata replaces the bespoke as_uri challenge parameter and /.well-known/lws-configuration path.

Authorization server discovery (RFC 9728)

A storage server generating a 401 Unauthorized response MUST send a WWW-Authenticate challenge containing:

  • resource_metadata (REQUIRED) — the URI of the storage server's protected resource metadata document, per [[!RFC9728]] Section 5.1;
  • realm (REQUIRED) — an absolute URI indicating the scope of protection. This value is what the client passes as the token-exchange resource parameter, and appears as the access token's aud. Under path alignment the realm is typically the storage root. A client MUST verify that the URI of the originating request is logically contained within the presented realm before using it (jeswr/lws-keycloak spec/authz.md §4.1, §7.5).
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="https://storage.example/alice/",
    resource_metadata="https://storage.example/.well-known/oauth-protected-resource/alice"

Per [[!RFC6750]] Section 3.1, the challenge carries an error parameter (e.g. error="invalid_token") only when the request presented an access token that the server rejected; a challenge answering a request that bore no credentials — as in the example above — MUST NOT include an error parameter.

The protected resource metadata document ([[!RFC9728]] Section 2) MUST include resource and authorization_servers (the trusted authorization servers — the storage server manages this canonical list), and MUST include the extension member jlws_storage_description: the URI of the storage description resource. When a proof-of-possession presentation profile is offered, the relevant metadata members (e.g. dpop_signing_alg_values_supported, and the DPoP-SK members defined by [[DPOP-SK]]) appear here ().

Authorization server metadata is served per [[!RFC8414]] at its registered well-known URI (/.well-known/oauth-authorization-server), and SHOULD include subject_token_types_supported listing the authentication suites accepted at the token endpoint.

The Working Draft mints a bespoke as_uri challenge parameter and an unregistered /.well-known/lws-configuration path with RFC 8414 shape. RFC 9728 (Proposed Standard, 2025) standardises exactly this pattern — a resource_metadata challenge parameter and a registered well-known document listing the resource's authorization servers. The editor's own prototype already reached for the same shape (the RECOMMENDED storage_metadata challenge parameter, jeswr/lws-keycloak spec/authz.md §4.1), so adopting RFC 9728 upgrades his design to the shipped standard rather than diverging from it. The realm parameter and its containment-verification rule are retained unchanged. Recorded as DECISIONS.md D1.

Token exchange

Per [[LWS10-CORE]] Authorization §4.3 (verbatim in requirements):

  • The authorization server's token endpoint MUST support urn:ietf:params:oauth:grant-type:token-exchange [[!RFC8693]].
  • The resource parameter is REQUIRED, MUST be an absolute URI, populates the access token's aud, and equals the challenge's realm. The authorization server MUST reject a resource identifying an unknown or untrusted storage.
  • The subject_token parameter is REQUIRED and carries a valid authentication credential; subject_token_type names its suite. The authorization server MUST validate all presented tokens before issuing.
  • Errors follow [[!RFC6749]] Section 5.2.

Access token

The issued access token MUST conform to [[!RFC9068]] (at+jwt) with:

  • sub — REQUIRED; absolute URI of the agent;
  • iss — REQUIRED; absolute URI of the authorization server;
  • client_id — REQUIRED; the client identifier;
  • aud — REQUIRED; exactly one value: the absolute URI supplied as resource. (This resolves the Working Draft's internal inconsistency between "exactly one value" validation and multi-audience examples. The constraint binds access tokens — where the replay risk lives — not authentication credentials: a suite's subject token, e.g. an OIDC ID token, may carry multi-valued aud as its suite specifies, and MUST NOT be rejected for that. Recorded as DECISIONS.md D11.)
  • exp — REQUIRED; lifetimes of 300 seconds or less are RECOMMENDED;
  • iat, jti — REQUIRED.

Rich authorization details (OPTIONAL)

Authorization servers MAY evaluate fine-grained requests and encode the decision in an authorization_details claim ([[!RFC9396]]) whose entries have type equal to https://w3id.org/jeswr/lws#AccessRequest and carry locations (absolute URIs within the token's aud), actions (from the action set of ), and optionally datatypes and purposes. Two modes exist, per the editor's prototype ("AS responds with either: yes, these actions are ok; OR AS responds with deferred policy evaluation" — jeswr/lws-keycloak spec/authz.md §1): approved (the AS attests the actions; the storage server enforces the encoded decision in addition to its own policy) or deferred (the AS passes the request through; the storage server evaluates it against its policy layer). A storage server MUST NOT treat an authorization_details claim as widening access beyond its own policy: the claim can only narrow.

Token presentation

Bearer (baseline, REQUIRED)

Clients present the access token in the Authorization header with the Bearer scheme [[!RFC6750]]. A JLWS server MUST accept Bearer presentation for every realm except one it has designated proof-of-possession-required (); that per-realm designation — advertised in the realm's resource metadata and its challenges — is the sole permitted exception to Bearer acceptance. The replay exposure of bearer semantics is bounded by the mandatory audience restriction and the recommended ≤300-second lifetime; the residual within-lifetime replay risk is analysed in .

Proof-of-possession profiles (OPTIONAL, negotiated)

A server MAY additionally offer sender-constrained presentation, advertised through its RFC 9728 protected resource metadata; a client discovers and opts in, and a client that does not opt in is unaffected. Two profiles are defined:

  • DPoP [[!RFC9449]] — advertised via dpop_signing_alg_values_supported; the token is bound at exchange time via dpop_jkt/cnf.jkt and each request carries a DPoP proof.
  • DPoP-SK [[DPOP-SK]] — the negotiated symmetric fast path: one DPoP proof establishes a short-lived session key; each request carries an RFC 9421 hmac-sha256 message signature with sliding-window replay protection. Its negotiation is already RFC 9728-based and composes here without change.

A server MAY, by deployment policy, require a proof-of-possession profile for a designated realm (e.g. medical or financial storages — [[LWS-UCS]] use cases). The signal is concrete, on both discovery surfaces: the realm's protected resource metadata MUST set dpop_bound_access_tokens_required: true ([[!RFC9728]] Section 2 — or, for DPoP-SK, the analogous required-member defined by [[DPOP-SK]]), and its 401 challenges MUST list only the required profile's scheme(s) (e.g. WWW-Authenticate: DPoP … per [[!RFC9449]] Section 7.1), omitting the Bearer scheme. When the member is absent or false, Bearer is accepted (with PoP optionally available per the advertised dpop_signing_alg_values_supported) — a client can therefore always distinguish Bearer-accepted, PoP optional from PoP required. By default, Bearer remains accepted.

The no-DPoP baseline is the editor's deliberate design ("The use of DPoP is no longer required" — jeswr/lws-keycloak spec/authn.md, migration notes; the Working Group design likewise: "all token audiences are restricted and tokens are not sender constrained (no DPoP)" — the Working Group's published design summary, public-lws-wg mailing list, Nov 2025). This specification keeps that baseline and adds PoP only as a negotiated option for deployments whose threat model needs it: within its ≤300-second lifetime a stolen Bearer token replays freely against its own storage, and the Working Draft's own security text enumerates theft mitigations that PoP closes. Costs nothing when off. Recorded as DECISIONS.md D9 (steer default 2).

Validation by the storage server

A storage server MUST validate a presented access token by, in order (jeswr/lws-keycloak spec/authz.md §4.4.2):

  1. Signature — verify the JWT signature against the authorization server's JWKS (from its RFC 8414 metadata); cache keys; support rotation; reject alg: none and unexpected algorithms.
  2. Issuer — iss MUST match a member of the storage server's explicit trusted-issuer allowlist ().
  3. Audience — aud MUST contain exactly one value, an absolute URI that logically contains the target resource. Logical containment is evaluated over parsed, normalized URIs ([[!RFC3986]] Section 6) — never by raw string prefix: the audience and the target MUST have the same origin (scheme, host, port), and the audience's path MUST equal the target's path or be an ancestor of it on complete /-delimited segment boundaries. (An audience of …/alice/ contains …/alice/notes/x but NOT …/alicemalicious; a raw prefix comparison would authorize sibling resources.) Multi-tenant deployments MUST scope acceptance to the current storage (the tenant-replay rule, spec/authz.md §7.5.1).
  4. Temporal — exp in the future, nbf (if present) passed, iat not in the future; clock skew tolerance of at most 60 seconds is RECOMMENDED; tokens with exp more than one hour ahead MUST be rejected.
  5. Proof of possession — when the token is PoP-bound (cnf present), the corresponding proof MUST be validated per the profile in use, and the token MUST NOT be accepted bare.

On validation failure the server responds 401 with an appropriate error parameter. If the token is valid but the operation is not allowed: 403 when the agent has some access to the resource, otherwise 404 to avoid disclosing existence ().

Client-side rules

  • A client MUST verify realm containment before requesting a token for it ().
  • A client MUST NOT present a token whose aud does not logically contain the target resource, MUST NOT present a token with more than one aud value, and SHOULD NOT share a token with any party other than the storage server it is scoped to.
  • A client SHOULD NOT proactively obtain tokens from an authorization server before receiving a 401 challenge for a resource in that realm — preventing the colluding-RS/AS probe described in (jeswr/lws-keycloak spec/authz.md §7.10).
  • Clients SHOULD cache tokens and reuse them proactively for further resources logically contained in the token's aud during its lifetime.

Access requests and grants

Access requests and grants are the interchange interface over the server's internal policy layer; the policy language a server enforces with is out of scope, matching the Working Group's resolution (London F2F Day 1; w3c/lws-protocol#143). The document model is a strict ODRL profile — the design of the editor's upstream PR #109, realised.

The ODRL access profile

  • Documents are JSON-LD using the ODRL context plus the JLWS context, and MUST declare profile equal to https://w3id.org/jeswr/lws/access-profile/odrl-1 (REQUIRED — per PR #109, an unprofiled ODRL document is not a conforming access document).
  • An access request is an odrl:Request; an access grant is an odrl:Offer issued by the storage controller (PR #109's mapping).
  • A document carries ODRL rules: permission, prohibition, and obligation ([[!ODRL-MODEL]]). Each rule MUST have exactly one action and exactly one target (singular per rule — PR #109; compose multiple rules for multiple actions/targets).
  • Actions: odrl:read, odrl:modify, odrl:delete, plus jlws:create and jlws:append defined by this profile with odrl:includedIn odrl:modify. The inclusion is one-directional: a grant of odrl:modify also permits the narrower jlws:create and jlws:append operations, but a create-only or append-only grant MUST NOT be widened to modify. An enforcement layer that does not understand a profile-defined action MUST NOT honour a rule granting it (fail closed — an unknown action grants nothing); such a layer still permits create/append operations under an odrl:modify grant, which is the deployment path for ODRL-core-only enforcement.
  • Targets: typed matchers — a DataResource (one resource), a Container (the container and, when jlws:recursive is true, its descendants), or a StorageResource (the whole storage), identified by URI, path-aligned per .
  • Assignee: a URI identifying the agent (or group, ); public access is expressed with http://xmlns.com/foaf/0.1/Agent ([[LWS10-CORE]] Access Requests).
  • Constraints: ODRL constraints; this profile defines the left operands odrl:purpose (via hasPurpose-to-odrl:purpose normalisation per PR #109), jlws:client, jlws:mediaType, jlws:resourceType, and odrl:dateTime. When multiple constraints are present, all MUST be satisfied.

  "uid": "https://storage.example/alice/.grants/2f6c",

Endpoints

Requests and grants live in ordinary JLWS containers advertised in the storage description as services of type AccessRequestService and AccessGrantService ([[LWS10-CORE]] Access Requests), each carrying conformsTo with the profile URI. An agent files a request by POST to the request container; the storage controller lists (GET), grants (POST to the grant container), and revokes (DELETE the grant resource). Servers SHOULD deliver notification of new requests and of grant creation/revocation to the affected agents' inboxes or subscriptions ().

Grants are records, not tokens

An agent does not present an access grant to a server as an authorization token ([[LWS10-CORE]] Access Requests). A grant records what the storage controller decided; the server MUST realise that decision in its internal enforcement:

  • Grant creation and revocation MUST be reflected in enforcement within a bounded, documented interval (SHOULD be immediate).
  • Derived views — container listings, type indexes, query results — MUST reflect revocation immediately (the live-authorization rule of [[LWS-SEARCHINDEX-ED]], generalised).
  • Grants are the authoritative interchange record; whether the server's policy engine treats them as its source of truth or as a projection is an implementation choice (the Working Group contention in w3c/lws-protocol#144, resolved here at the interface: the record binds observable behaviour, not internals).

Grants as verifiable credentials (OPTIONAL)

A storage controller MAY issue an access grant additionally as a W3C Verifiable Credential [[VC-DATA-MODEL-2.0]] whose credentialSubject embeds the ODRL grant — a verifiable, auditable, revocable consent receipt satisfying [[LWS-UCS]] requirement 18. Revocation of the grant MUST be reflected in the credential's status mechanism.

Delegation (OPTIONAL extension)

Servers MAY support delegated grants: a grant whose assignee is itself authorised to issue a narrower grant, forming an ODRL delegation chain. Chain semantics — monotonic narrowing (a delegated grant MUST NOT exceed its parent in actions, targets, constraints, or duration), expiry/revocation propagation, and cycle rejection — follow the delegation-chain model implemented in [[SOLID-ODRL]]. This serves the sub-delegation requirement the Working Group's model leaves unserved ([[LWS-UCS]] requirements 6 and 14).

Groups (OPTIONAL extension)

Servers MAY advertise a GroupService: group membership documents (JSON-LD, listing member agent URIs and, for hierarchies, member group URIs) that may appear as grant assignees. Group resolution happens at enforcement time; membership changes MUST take effect like grant changes (). Cyclic group membership MUST be rejected. Serves [[LWS-UCS]] requirement 5 (including hierarchies).

Discovery: the storage description

Every storage publishes one self-describing document — the storage description resource — enumerating its identity, verification keys, services, capabilities, and protocol conformance. This is the feature-detection surface for everything optional in this specification and its companions.

Binding

All responses to GET and HEAD requests targeting storage resources MUST include a Link header targeting the storage description with rel="https://w3id.org/jeswr/lws#storageDescription" (alias rule of applies), and servers SHOULD include it on 401 responses to guide unauthenticated clients ([[LWS10-CORE]] Discovery). The document MUST be readable by any agent that can read the storage root, and MAY be public. When the storage description is not public and the storage delivers signed webhook notifications (), the verificationMethod key material that notification keyid values resolve to MUST nonetheless be fetchable without storage credentials — either by making the storage description itself public, or by publishing a public key-only [[!CID-1.0]] document and using its verification-method URIs as the keyid values. A webhook receiver is not a storage client and cannot be assumed to hold read access; a signature it cannot verify is a signature it will be tempted to skip.

Data model (CID-shaped)

The storage description is a JSON-LD document with:

  • id — REQUIRED; the storage's URI;
  • type — REQUIRED; includes Storage;
  • conformsTo — REQUIRED; an array of protocol version URIs the storage implements, e.g. https://w3id.org/jeswr/lws/protocol/core/1.0 and one URI per implemented companion profile. (Convention adopted from fedreg:StorageDescription [[FEDREG]], which advertises storage spec versions the same way — federations can index JLWS storages natively.)
  • verificationMethod — OPTIONAL (REQUIRED for notification servers); the storage's signing keys as [[!CID-1.0]] verification methods — webhook signatures resolve keyid into this set ();
  • service — REQUIRED; a set of {id?, type, serviceEndpoint, …} entries, including one of type StorageDescription whose serviceEndpoint is the document's own URL;
  • capability — OPTIONAL; a set of {id?, type, …} entries from or extension URIs.

Consumers MUST ignore unrecognised capability and service types (forward compatibility). The document MUST validate as a controlled identifier document when verificationMethod is present.

The Working Draft's storage description has capability/service but no keys, no protocol-version advertisement, and only unregistered example capability types (https://feature.example/…). The CID remodel is the Working Group's own live direction (w3c/lws-protocol#183, discussed 2026-06-29; the Ghent F2F artifact already has this shape, and the notifications Editor's Draft already resolves webhook keys from it) — this specification simply lands it, adds conformsTo, and mints a real capability registry. Recorded as DECISIONS.md D5.

Notifications

A notification server lets agents subscribe to resource changes instead of polling. The subscription API and envelope adopt the Working Group's notifications Editor's Draft [[LWS-NOTIFICATIONS-ED]]; the webhook binding is adopted verbatim; the SSE and WebSocket bindings are added under the same subscription API.

The Working Group resolved to specify server-to-server webhooks first and defer server-to-client channels to a separate work item (London F2F Day 2 resolutions), leaving browser clients with nothing; the Use Cases document names "real-time (e.g., push/SSE) or via queued channels" explicitly ([[LWS-UCS]] requirement 7). This specification ships both deferred bindings, addressing the SSE-authentication concern raised in that discussion head-on (). Recorded as DECISIONS.md D6; SSE-first vs WebSocket-first is steer question 6 in OPEN-QUESTIONS.md (this draft ships both).

Subscription API

  • Subscriptions are created by authenticated POST to the NotificationService endpoint with a JSON-LD body: type — one of the advertised subscription types; topic — REQUIRED array of resource URIs; expires — OPTIONAL datetime; plus binding-specific members below.
  • The server MUST reject a subscription whose agent lacks read access to every topic ([[LWS-NOTIFICATIONS-ED]]).
  • A subscription to a container is recursive: events for the container and all descendants.
  • Authorization is enforced at subscription and delivery time: if the subscriber's read access to a topic is revoked, deliveries for it MUST cease immediately.
  • The response returns the subscription resource (its URI allows GET inspection and DELETE cancellation) and binding-specific connection details.

Envelope

A notification is a JSON-LD document of type Notification wrapping an Activity Streams 2.0 activity [[AS2-VOCAB]] — Create, Update, or Delete — whose object carries the affected resource's id and type only. Notifications MUST NOT include resource content. published is REQUIRED; actor MUST be omitted unless the subscriber is authorised to learn it (privacy default of [[LWS-NOTIFICATIONS-ED]]). Servers MAY batch multiple activities per delivery.

Webhook binding (WebhookSubscription)

Per [[LWS-NOTIFICATIONS-ED]], adopted verbatim:

  • The subscription carries inbox — the HTTPS endpoint to deliver to. Delivery is HTTP POST of the envelope.
  • Deliveries MUST be signed with HTTP Message Signatures [[!RFC9421]] covering at least @method, @scheme/@authority/@path (or @target-uri), content-type, and content-digest ([[!RFC9530]]).
  • The signature's keyid MUST be a URL-with-fragment resolving into the storage description's verificationMethod set (); receivers MUST verify the signature before processing, fetching the storage description subject to .
  • Delivery-side SSRF discipline applies to the storage server's outbound POST (): inbox URIs are untrusted input.

SSE binding (SseSubscription) — added

  • The subscription response includes source: the event-stream URL. The client GETs it with Accept: text/event-stream.
  • Stream authentication — the server MUST support at least one of, and the subscription response states which apply:
    • Header mode: the stream request carries Authorization with a valid access token for the storage (re-challenged per on expiry: the server ends the stream with a terminal event of type reauthenticate, and the client reconnects with a fresh token);
    • Capability-URL mode (for EventSource clients that cannot set headers): source is a single-subscription, unguessable (≥128 bits of entropy), expiring URL. It MUST be bound to the subscription, MUST expire no later than the subscription, MUST be revoked with it, and MUST only be issued over TLS. This addresses the SSE-authentication objection recorded in the Working Group's channel discussion.
  • Every event MUST carry an id field. On reconnection with Last-Event-ID, the server MUST either replay all events after that id (it MUST retain events for a documented retention window) or, when the id has aged out, send a control event of type reset telling the client to re-synchronise state by reading the topics.
  • Delivery-time authorization applies per event ().

WebSocket binding (WebSocketSubscription) — added

  • The subscription response includes receiveFrom: a wss: URL — a capability URL with the same binding, entropy, expiry, and revocation requirements as SSE capability-URL mode. The shape intentionally mirrors [[WEBSOCKETCHANNEL2023]] (receiveFrom) so existing Solid notification clients port with minimal change.
  • Each WebSocket message is one envelope (), serialised as a text frame.
  • Servers SHOULD ping periodically; clients re-subscribe (obtaining a fresh receiveFrom) after close or expiry. There is no replay on this binding; clients needing gapless delivery use SSE or webhooks.

Content transformation (capability hook)

A server MAY offer server-side transformation between representations of a data resource, advertised per source media type with the ContentNegotiation capability (): source (a media type), target[] (media types derivable from it), and optionally profile — a URI pinning the transformation semantics — and normalizes. A server MUST NOT perform transformation it does not advertise, and clients MUST NOT expect any transformation that is not advertised: absent the capability, a data resource has exactly its stored representation.

The stored byte stream remains authoritative regardless of any advertised transformation: reads in the stored media type MUST return the stored bytes exactly (unless normalizes is declared), and write preconditions evaluate against the authoritative representation's entity-tag. The precise multi-representation ETag, precondition, and round-trip rules — and the RDF transformation semantics that make a JLWS storage RDF-aware — are defined by the companion JLWS RDF Content Transformation Profile (profile URI https://w3id.org/jeswr/lws/transform/rdf-1). This layering is the requirements-level "Personal Data Projection" of [[LWS-UCS]] requirement 33.

Query services (service slots)

Query is out of the substrate core and defined by a companion specification (planned); this section reserves the service types and fixes the invariants any query service MUST satisfy, generalising [[LWS-SEARCHINDEX-ED]]:

  • TypeIndexService — a paginated listing of the distinct resource types present in the storage; TypeSearchService — filtered search over type and descriptive link relations, with equivalent GET and POST forms, and optionally an HTTP QUERY binding ([[HTTP-QUERY]]) as an additional form — QUERY MUST NOT be the only binding (respecting the deployment objection recorded in the 2026-06-29 minutes).
  • SparqlQueryService — access-controlled SPARQL over RDF-readable resources, available only where the RDF Content Transformation profile is on; semantics (per-solution authorization filtering) follow [[AC-SPARQL]]. This is the answer to [[LWS-UCS]] requirement 15, which the type-search ceiling deliberately does not meet (content query was "out of scope but not prohibited" — Ghent F2F minutes).
  • Invariants (normative for any advertised query service): results MUST be filtered by the requesting agent's live authorization on every request (revocations take effect immediately); counts MUST reflect only the caller-visible view; index derivation MAY be eventually consistent but authorization filtering MUST NOT be; and a query service MUST NOT reveal the existence of resources the agent cannot read ().

Versioning (capability slot)

Versioning is a companion (planned), reserved here as the Versioning capability: version history discovered via rel="version-history" ([[!RFC5829]]) and point-in-time reads via Memento Accept-Datetime ([[!RFC7089]]) — standards-based and cheap, serving [[LWS-UCS]] requirement 30. Version resources are read-only and subject to the same authorization as their originals at read time.

Security considerations

The recommendations of Best Current Practice for OAuth 2.0 Security [[!RFC9700]] apply to this entire specification. This section adds JLWS-specific requirements; unlike its Working-Draft counterpart it is normative where marked.

Transport

All protocol interactions MUST use TLS; clients MUST validate server certificates (service identity) per [[!RFC6125]] and follow the TLS configuration recommendations of BCP 195 ([[!RFC9325]]). Tokens MUST NOT appear in URLs, browser storage accessible to untrusted scripts, or logs (log truncated/hashed values if needed). Capability URLs () are the deliberate, constrained exception to the no-secrets-in-URLs rule: single-purpose, expiring, revocable, TLS-only.

Token security

  • Bearer tokens are theft- and replay-exposed within their lifetime; the mandatory mitigations are audience restriction to exactly one storage-containing URI and lifetimes of ≤300 seconds (RECOMMENDED). Deployments handling high-value data SHOULD offer — and MAY require — a proof-of-possession profile ().
  • Storage servers MUST maintain an explicit allowlist of trusted authorization-server issuers and reject all others: a malicious AS can otherwise mint arbitrary access.
  • Multi-tenant instances MUST NOT accept a token whose aud is scoped outside the current storage (tenant-replay isolation, jeswr/lws-keycloak spec/authz.md §7.5.1).
  • Servers MAY maintain a jti replay cache (bounded, sized against DoS) for single-use token policies.
  • Authentication credentials with unrestricted audiences MUST NOT be presented to untrusted authorization servers; clients SHOULD obtain audience-restricted credentials (RFC 8707 resource indicators, or token exchange) first.

Colluding resource/authorization servers

A malicious storage server can advertise a realm broader than its own scope with a colluding authorization server; a client that proactively fetches tokens for that realm leaks its intent to access other storages within the over-broad scope. Hence the client rules of : verify realm containment against the originating request, and do not fetch tokens before a 401 from the actual target (jeswr/lws-keycloak spec/authz.md §7.10).

Server-side request forgery (normative)

A JLWS server and authorization server make outbound requests to attacker-influenced URLs: CID/subject dereferencing (), JWKS and metadata fetches, client-ID metadata fetches, webhook inbox delivery (), and storage-description fetches by webhook receivers. Every such fetch MUST apply all of the following:

  • Scheme: https only.
  • Address policy: resolve the hostname, then reject loopback, private-range, link-local, unique-local, and cloud-metadata addresses (IPv4 and IPv6, including IPv4-mapped/compatible IPv6 forms) — decided on the resolved address, not the hostname string.
  • Rebinding: connect to the validated resolved address (DNS pinning); do not re-resolve between validation and connection.
  • Redirects: do not follow redirects automatically; if followed, each hop MUST re-pass the full policy.
  • Caps: enforce response-size and time limits; bound concurrent outbound fetches per principal.
  • No credential forwarding: outbound fetches MUST NOT attach the inbound request's credentials.
The Working Draft has no SSRF text, yet its design (CID dereference on validation, webhook delivery to subscriber-supplied inboxes) is a textbook SSRF surface for a server sitting on cloud metadata endpoints and internal networks. These requirements mirror the policy proven across the editor's server implementations. Recorded as DECISIONS.md D10.

Oracle freedom (normative)

  • If an agent has no permitted operation on a resource — and is not otherwise authorized to learn that it exists (e.g. through a readable parent listing) — the server MUST respond 404 (not 403), indistinguishable from a genuinely absent resource: a 403 here is precisely the existence oracle the error-response rule below forbids. 403 is reserved for agents with partial access, i.e. agents already entitled to know the resource exists (jeswr/lws-keycloak spec/authz.md §7.12).
  • Container listings, totalItems, query results, and notification streams MUST expose only what the requesting agent can read (, ).
  • Servers MUST NOT enumerate, in any discovery surface, the set of link relations or types they index (no discovery oracle — [[LWS-SEARCHINDEX-ED]]).
  • Error responses (including problem details) MUST NOT leak the existence, size, or type of resources the agent cannot read.

Web-platform integration

Servers supporting browser clients MUST validate Origin and use restrictive CORS policies; wildcard credentialed CORS MUST NOT be used. Servers SHOULD emit X-Content-Type-Options: nosniff on data-resource responses and treat stored content as untrusted when serving to browsers (e.g. Content-Disposition or sandboxing policy for active content).

Privacy considerations

  • Minimal disclosure: authorization servers SHOULD issue tokens containing only what authorization needs; issuers of authentication credentials likewise (jeswr/lws-keycloak spec/authz.md §6).
  • Pseudonymous presentation: when pseudonymous subject identifiers are used, clients SHOULD request batch issuance and use each token once, preventing the storage server from correlating requests by identifier; an authorization server SHOULD NOT issue the same pseudonymous identifier twice (spec/authz.md §6.1 — note IP-address and policy-shape correlation remain).
  • Suite advertisement: subject_token_types_supported is SHOULD-level because a self-hosting individual may not wish to advertise their authentication technology (the Working Draft's own rationale).
  • Notifications: the envelope omits actor by default and never carries content (); webhook inboxes learn activity timing — subscribers should be told this when registering third-party inboxes.
  • Logging: implementations SHOULD NOT log full tokens, credentials, or capability URLs.
  • Storage descriptions may be public; publishers should treat capability and service enumeration as disclosure and MAY scope the document's readability.
